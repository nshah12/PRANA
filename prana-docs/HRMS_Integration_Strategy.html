Why three modes? Indian enterprises vary widely in HRMS maturity. Large BFSI and MNCs run SAP/Workday with full APIs. Mid-market uses Darwinbox/Keka. SMEs use greytHR/Spine or nothing. The three modes ensure PRANA is accessible at every tier without changing the core pipeline.

1

Direct HRMS Integration

PRANA connects directly to the employer's HRMS via its REST API or webhook.
Pull (Primary)

Scheduled Pull

HRMSPullWorkflow (Temporal) runs per-tenant on schedule. Delta pull via last_pulled_at cursor. Authenticate → list employees → list new docs → download → ingest.

Push

HRMS-Initiated Push

HRMS posts document payload to POST /v1/ingest/hrms. HMAC-SHA256 signed. Immediate 202 response. Pipeline runs async via Kafka.

Webhook

Event Webhook

HRMS fires event on new document. PRANA receives at POST /v1/hrms/webhook/{connector_type} → targeted pull of that one document. Real-time, no polling.

HTTP Handler Contract (NEVER violate)

validate HMAC signature → S3.put(document_bytes) → INSERT document row → kafka.publish("prana.ingest.events", DOC_INGESTED) → 202 Accepted
// NO audit writes · NO workflow starts · NO notification dispatch

Connector Abstraction Layer

class HRMSConnector(Protocol): async def authenticate(credentials: dict) → str # bearer token async def list_employees(since: datetime) → list[dict] # delta async def list_documents(emp_id, since) → list[dict] async def download_document(doc_url, token) → bytes

2

Shared Location Pickup

HRMS or payroll system drops files to a shared location. PRANA monitors, picks up, maps, and ingests automatically.

Ideal for HRMS systems with weak APIs (Spine, Beehive), on-prem payroll systems (Tally, custom Java apps), or enterprise IT policies that prohibit direct API access from SaaS vendors.

Two Mapping Strategies

Strategy A — Preferred

Metadata Manifest

HRMS drops a JSON or CSV manifest alongside document files. Manifest contains explicit employee_id, doc_type, doc_period. Deterministic mapping — no AI needed. Fast and reliable.

Strategy B — Fallback

AI Mapping

No manifest present. FileMappingWorkflow invokes Stage 03 LLM (Qwen) to extract doc type and employee identity signals from the document. Stage 04 resolution matches to employee_master. Slower but fully automated.

SharedLocationMonitorWorkflow (Temporal)

// S3 — preferred: near real-time via S3 Event Notifications
S3 Event Notification → SNS → SQS trigger → SharedLocationMonitorWorkflow

// SFTP / GDrive / SharePoint / Azure Blob — scheduled poll
Temporal timer (configurable per tenant) → list new files → download → manifest check
→ if manifest: deterministic map → ingest
→ else: FileMappingWorkflow (AI) → ingest

3

Template Wizard — PRANA as Document Source

Employer builds document templates in the PRANA Portal. PRANA generates the PDFs. No HRMS required.

Targets SMEs and startups that have no HRMS, or employers who want standardized document formats. PRANA becomes the system of record for generated documents.

🔒 Privacy Contract — STRICTLY enforced in Template Wizard
Payroll data (salary amounts, PAN) lives only in memory during PDF generation. After the PDF is streamed to S3, all raw figures are zeroed in memory. is_self_upload = TRUE flags Portal-generated docs. Stage 03–05 AI pipeline runs identically — extracts insights, never stores raw figures. Violation = privacy breach.

Generation Flow

OA-Admin selects template → uploads payroll CSV (in-memory only)
→ Jinja2 renders HTML per employee
→ WeasyPrint converts HTML → PDF bytes (in-memory)
→ raw salary amounts zeroed in object
→ PDF streamed to S3 (never disk)
→ INSERT document row (is_self_upload=TRUE)
→ kafka.publish(DOC_INGESTED) → 202
// Standard AI pipeline (Stage 01–06) runs from here — unchanged
