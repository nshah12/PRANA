Kafka — AWS MSK Cluster Upgrade
Today (before)
  • 5 topics, 12 partitions each
  • KRaft mode, MirrorMaker 2 bidirectional
  • Single consumer group per topic
  • No schema registry
  • No dead-letter topics
  • No topic-level retention config
Target (after)
  • 21 topics, partitions tuned by throughput
  • High-throughput (auth, vault, audit): 48 partitions
  • Medium (pipeline, analytics): 24 partitions
  • Low (compliance, platform): 12 partitions
  • Schema Registry (Confluent / AWS Glue)
  • Dead Letter Topic per domain (e.g. prana.auth.events.dlq)
  • Topic-level retention: audit=7yr, notif=30d, pipeline=90d, others=7d
  • Consumer lag alarms via CloudWatch
  • Idempotent producers on all services
Partition Key Rules
  • Auth events → user_id
  • Ingest / Pipeline → document_id
  • Tenant events → tenant_id
  • Vault access → employee_uuid
  • Compliance → employee_user_id
  • Security → tenant_id
  • Notifications → recipient_id
  • Audit sink → tenant_id
  • Cache invalidation → cache_key prefix

Redis — ElastiCache Cluster Mode Upgrade
Today (before)
  • Global Datastore — single shard, CRDT active-active
  • 4 namespace groups: pan, share, vault, revoked
  • No cluster sharding
  • No rate limiting in Redis
  • No distributed locks
  • Config fetched from DB on every activity
  • API key validated from DB on every ingest
  • No dropdown caching
Target (after)
  • Redis Cluster Mode — 6 shards × 2 replicas each
  • 16,384 hash slots distributed across shards
  • Shard 0–1: Hot (auth, sessions, rate limits, locks, SSE)
  • Shard 2–3: Warm (config, manifests, profiles, dropdowns)
  • Shard 4: Analytics + leaderboards
  • Shard 5: Streams (audit buffer, access log buffer)
  • R/W split: reads → replicas, writes → primary
  • Global Datastore CRDT across ap-south-1 + ap-south-2
  • Redis Streams for audit event buffer
  • Keyspace notifications for TTL expiry callbacks
Invalidation Rules
  • Config change → publish CONFIG_INVALIDATE to prana.cache.invalidation → all pods DEL config:*
  • API key revoke → publish APIKEY_INVALIDATE → DEL apikey:{hash}
  • Manifest upsert → DEL tenant:{id}:manifest:*
  • Employee exit → DEL emp:profile:{id}
  • OA elevation end → DEL oa:elevation:{id}
  • Consent withdraw → DEL emp:consent:{id}:*, immediate
  • Tenant suspend → DEL tenant:{id}:* (all keys)
Two Highest-Value Wins
  • Config cache — platform_config and tenant_config read by every Temporal activity. With 53 workflows × 11 queues, hundreds of DB reads/min eliminated with 15min TTL cache.
  • API key cache — every HRMS push validates API key from DB before touching S3. With 1 lakh orgs, millions of DB reads/day → 5min Redis cache with instant invalidation on revoke.
  • Access log buffer — Redis Stream buffers document_access_log writes, flushes to DB every 30s. Zero latency added to document view.
