What is Agentic AI in PRANA?

Agents are Temporal workflows that call LLMs iteratively, invoke external tools, and pause for human approval — not single-shot LLM calls.

🤖

Multi-Step Reasoning

Each agent decomposes a complex task into sub-steps: gather evidence, analyse, decide, act. No single-shot prompting — agents iterate until confident or escalate.

🔧

Tool-Using

Agents call real tools: EPFO API, TRACES cross-check, YugabyteDB queries, Kafka publish, Temporal signals. LLM decides which tool to call next based on intermediate results.

⏱️

Temporal-Backed

Every agent run is a named Temporal workflow — durable, retryable, replayable. Long-running agents (e.g. grievance resolution over 30 days) survive restarts automatically.

🧑

Human-in-the-Loop

Agents pause at decision points needing human approval. Temporal signals resume the workflow after OA-Admin acts. No polling — push-based via SSE → browser.

🔒

Privacy Contract Enforced

Agents follow the same LLM privacy contract as the pipeline: raw ₹ salary and PAN are never in LLM output, never stored, never surfaced in any agent response.

📡

Event-Driven Triggers

Agents start from Kafka events (document ingested, grievance filed, vault gap detected) — not from cron jobs or polling loops. Kafka consumer → WorkflowConsumer → Temporal.

Core Agentic Principles PRANA Follows

Agents never bypass the HTTP handler Kafka contract — no direct DB writes from agent LLM output
Every agent action is audit-logged via AuditConsumer — agents cannot skip audit trails
Agent LLM receives document context only if tenant_id and employee_user_id match — no cross-tenant data leakage
Tool calls use the same asyncpg pool as API handlers — no separate DB connections for agents
Agent max steps = 12 per run — hard cap prevents runaway loops; escalates to OA-Admin if not resolved
All agent outputs go through privacy filter before storage or surfacing — same as Stage 05 pipeline

01

Document Verification Agent

Cross-checks Form 16 vs TRACES · PF slips vs EPFO UAN passbook
Trigger Employee submits Form 16 or PF slip via mobile app; or OA pushes batch of salary slips

Agent Steps

  • 1
    Extract structured data from document (OCR + Qwen LLM) — TAN, PAN-token, employer name, amounts
  • 2
    Derive TRACES query parameters from extracted TAN + PAN-token + FY
  • 3
    Call TRACES API / scraper — pull employer TDS deduction certificate
  • 4
    LLM diff step: compare extracted amounts vs TRACES figures (insight-level, no raw ₹ stored)
  • 5
    For PF slips: call EPFO UAN passbook API with employer credentials from tenant_config
  • 6
    Produce verification_result: MATCH | MISMATCH | UNVERIFIABLE with reason

Tools Used

Qwen2.5-14B (extract) TRACES API EPFO UAN API YugabyteDB (document) S3 (document bytes) Kafka (audit event) Temporal (durable run)

⚠ Human-in-the-Loop Points MISMATCH result → OA-Operator notified via Portal bell + email. OA must acknowledge or dispute before document status changes to VERIFIED_FAILED. Employee notified of outcome only after OA action.
✓ Output document.verification_status updated · audit_event written · employee push notification (insight only: "Your Form 16 verification is complete") · OA exception report if MISMATCH
Complexity: High Task Queue: prana-pipeline Workflow: DocumentVerificationWorkflow

Career Navigation Agent

Agent Steps

  • 1
    Retrieve employee career timeline from Qdrant (RAG over all career documents)

    Extract: current designation, years of experience, skill signals from offer letters + appraisals

  • 5
    Privacy filter: remove any specific salary figures from plan output

Tools Used

Llama-3.1-8B (RAG plan) bge-m3 (embeddings) Qdrant (career RAG) YugabyteDB (employee_master) Kafka (analytics event)

⚠ Human-in-the-Loop Points Fully autonomous — no human approval needed. Employee can save, share, or dismiss the plan. Saved plans surface in vault under "Career Planning" tab.
✓ Output Structured career plan in chat · saved as career_plan in vault · milestone reminders scheduled as Temporal timers · analytics event for plan generation rate

03

Compliance Audit Agent

Tenant-scoped scan · finds missing required docs · generates exception reports for CHRO/CFO
Trigger Monthly Temporal timer OR CHRO clicks "Run Compliance Audit" in Portal

Agent Steps

  • 1
    Load tenant's required document matrix from tenant_config (which doc types are mandatory for which employee grades)
  • 2
    Query all active employees in tenant: cross-join with document table to find gaps
  • 3
    Classify gaps: CRITICAL (Form 16 missing), WARNING (PF slip > 3 months old), INFO (offer letter missing)
  • 4
    LLM drafts human-readable exception report with risk classification per employee cohort
  • 5
    Publish exception_report_created event to Kafka → NotifConsumer emails CHRO/CFO
  • 6
    Store report as document in tenant vault with doc_type = COMPLIANCE_REPORT

Tools Used

Llama-3.1-8B (report draft) YugabyteDB (employee + document) Kafka (notification event) S3 (report storage) Temporal timer (schedule) Redis (vault completeness cache)

⚠ Human-in-the-Loop Points CHRO must approve before report is shared with HRMS partner. CFO approval required if financial document gaps exceed threshold in tenant_config. Approval via Portal → Temporal signal resumes workflow.
✓ Output Exception report PDF in tenant vault · CHRO/CFO email with summary · Portal dashboard badge updated · OA-Operator action list per employee gap
Complexity: Medium Task Queue: prana-analytics Workflow: ComplianceAuditWorkflow
04

Payroll Anomaly Agent

Detects CTC jumps without appraisal · duplicate PAN tokens across tenants · suspicious patterns
Trigger New salary slip ingested via HRMS push → DOC_INGESTED Kafka event consumed by WorkflowConsumer

Agent Steps

  • 1
    Retrieve employee's salary insight history (insight-level only, no raw ₹ figures) from document table for this pan_token
  • 2
    LLM compares salary band signals: detect significant jump vs previous month without corresponding appraisal document
  • 3
    Check pan_token across tenants: detect if same employee has active employment at competing tenants (cross-tenant dedup)
  • 4
    Statistical anomaly check: flag if salary insight is > 2 standard deviations from employee's peer cohort band
  • 5
    If anomaly detected: classify severity (HIGH/MEDIUM/LOW) and reason code
  • 6
    Publish anomaly_detected event → AuditConsumer logs → NotifConsumer alerts CFO/CISO

Tools Used

Qwen2.5-14B (pattern detect) YugabyteDB (document history) Redis (pan_token identity cache) Kafka (anomaly event) Temporal (durable analysis)

⚠ Human-in-the-Loop Points HIGH severity anomalies require CISO + CFO acknowledgement before alert is escalated to HR. Cross-tenant duplicate PAN detection escalates to Portal Admin (platform level, not tenant level).
✓ Output Anomaly record in audit_event · CFO/CISO portal notification · severity classification · recommended action (REQUEST_APPRAISAL_DOC, FLAG_FOR_REVIEW, ESCALATE_TO_ADMIN)
Complexity: High Task Queue: prana-pipeline Workflow: PayrollAnomalyWorkflow
05

HRMS Reconciliation Agent

Identity resolution when HRMS pushes unknown employee PAN token
Trigger HRMS push contains employee whose pan_token does not match any employee_master row in tenant

Agent Steps

  • 1
    Attempt fuzzy match on employee name + DOB + UAN from HRMS payload against existing employee_master rows
  • 2
    If fuzzy match found (score > 0.85): propose identity merge → queue for OA-Operator review
  • 3
    If no match: check if pan_token exists in other tenants (cross-org employee) → propose inter-tenant link
  • 4
    LLM drafts reconciliation recommendation with evidence: "Likely same employee based on name+UAN match at 92% confidence"
  • 5
    OA-Operator approves or rejects via Portal — Temporal signal resumes workflow
  • 6
    On approval: create employee_master row, link pan_token, retry original document ingestion

Tools Used

Qwen2.5-14B (identity reason) YugabyteDB (employee_master) Redis (pan_token identity cache) EPFO UAN verify Kafka (retry ingest event) Temporal signal (OA action)

⚠ Human-in-the-Loop Points ALL identity merge decisions require OA-Operator explicit approval. Agent never auto-merges identities. If OA rejects: document is held in EXCEPTION state and HRMS partner is notified to correct employee data.
✓ Output New employee_master row created · pan_token linked · document retry queued · audit trail of identity resolution decision · HRMS partner notified of outcome
Complexity: High Task Queue: prana-ingest Workflow: HRMSReconciliationWorkflow

Vault Gap Agent

Scans vault completeness · pings OA-Operator with prioritized missing doc list
Trigger Weekly Temporal timer per tenant OR new employee joins (employee_created event via Kafka)

Agent Steps

  • 1
    Load vault completeness requirements from tenant_config: required docs per grade, per employment type
  • 2
    Query document table for each active employee: what's present vs what's required
  • 3
    Score completeness per employee (0–100%) and per tenant overall
  • 4
    Prioritize gaps: critical docs missing for new joiners (< 30 days) ranked highest
  • 5
    LLM generates OA-Operator action list in natural language: "3 employees missing Form 16 for FY 2024-25 — push deadline is 31 July"
  • 6
    Update Redis vault completeness cache: vault:{pan_token} = completeness JSON

Tools Used

Llama-3.1-8B (action list) YugabyteDB (document + employee) Redis (vault completeness) Kafka (notification event) Temporal timer (weekly)

⚠ Human-in-the-Loop Points Fully autonomous scan and notification. OA-Operator can snooze individual alerts for 7 days via Portal. Snooze is a Temporal timer signal — no cron job needed.
✓ Output Portal OA-Operator dashboard updated with gap list · email digest to OA-Operator · completeness score per employee updated in Redis · tenant health score in analytics
Complexity: Low Task Queue: prana-analytics Workflow: VaultGapWorkflow

Grievance Resolution Agent

DPDP grievance → autonomously gathers evidence → drafts resolution under 48h SLA
Trigger Employee submits grievance via mobile or Portal: POST /dpdp/grievance → GrievanceWorkflow starts

Agent Steps

  • 1
    Parse grievance text: classify type (DATA_ACCESS, INCORRECT_DATA, PROCESSING_OBJECTION, ERASURE_DELAY)
  • 2
    Gather evidence autonomously: query document_access_log, consent_log, audit_event for employee's recent activity
  • 3
    LLM drafts auto-acknowledgement (within 48h per DPDP Act) with evidence summary
  • 4
    For DATA_ACCESS grievances: pull share token history, watermark logs, access_channel breakdown
  • 5
    Draft resolution recommendation for Grievance Officer with all evidence collated
  • 6
    Grievance Officer reviews draft in Portal — approves/modifies before sending to employee

Tools Used

Qwen2.5-14B (classify + draft) YugabyteDB (audit_event, access_log, consent_log) Kafka (notification event) Temporal timer (30-day SLA) AWS SES (acknowledgement email)

⚠ Human-in-the-Loop Points Grievance Officer must review and approve resolution before sending. Auto-escalates to Portal Admin if Grievance Officer doesn't act within 7 days. Employee receives auto-ack immediately — full resolution within 30 days per DPDP Act.
✓ Output DPDP-compliant acknowledgement within 48h · evidence bundle for Grievance Officer · resolution letter drafted · audit trail in audit_event · closure email to employee after approval
Complexity: High Task Queue: prana-compliance Workflow: GrievanceResolutionWorkflow

Tenant Onboarding Agent

Guides OA through KEK provisioning · API key setup · first document push · validation
Trigger Portal Admin creates new tenant → TenantOnboardingWorkflow starts automatically

Agent Steps

  • 1
    Provision tenant KEK in AWS KMS (ap-south-1): create customer-managed key, tag with tenant_id
  • 2
    Generate API key pair for HRMS integration: store key_hash in api_key table, send signing_secret securely to OA-Admin email
  • 3
    Guide OA through Platform Config via conversational UI: required_doc_matrix, grievance_officer, notification channels
  • 4
    Validate HRMS integration: send test payload to HRMS push endpoint, verify HMAC signature roundtrip
  • 5
    Push onboarding document checklist to Portal: "5 things to do before your first employee document arrives"
  • 6
    Mark onboarding complete: update tenant.onboarding_status = COMPLETE, publish onboarding_complete event

Tools Used

AWS KMS (KEK provision) YugabyteDB (tenant, api_key, platform_config) Llama-3.1-8B (guide text) Kafka (onboarding events) AWS SES (API key email) Temporal (durable onboarding)

⚠ Human-in-the-Loop Points OA-Admin must complete each config step in Portal — agent waits via Temporal timer (up to 7 days per step). Portal Admin can view onboarding progress and nudge OA. HMAC test must pass before agent marks onboarding complete.
✓ Output KEK provisioned in KMS · API key delivered securely · tenant_config populated · onboarding checklist in Portal · tenant ready to receive HRMS documents
Complexity: Medium Task Queue: prana-admin Workflow: TenantOnboardingWorkflow
🔒

Privacy Contract Applies to All Agents

Every agent that processes employee documents must follow the same LLM privacy contract as the AI pipeline:

  • LLM receives full document context for reasoning — this is required for accuracy
  • LLM output (agent decision, recommendation, draft) contains insights only — no raw ₹ salary figures, no PAN, no NIK
  • Agent-generated content passes through the same privacy filter as Stage 05 of the pipeline before storage or delivery
  • Cross-tenant queries use only pan_token (HMAC output) — never plaintext PAN in any agent query or log

How Agents Fit the Stack

Agents are Temporal workflows that call LLM activities. They consume Kafka events and publish results back to Kafka — no special infrastructure.

Agent Execution Flow

Trigger Event Kafka consumer / HTTP / Temporal timer
→
WorkflowConsumer Starts Temporal workflow
→
Agent Workflow Temporal: durable, replayable
▼
LLM Activity Qwen / Llama via LLMClient
→
Tool Activities DB / S3 / EPFO / TRACES / KMS
→
HITL Pause (if needed) Temporal.waitForSignal()
▼
Privacy Filter Strip raw ₹ / PAN from output
→
Kafka Publish audit / notification / analytics
→
Consumers Act AuditConsumer · NotifConsumer · SSEFanout
Key design principle: Agents call LLM as a Temporal activity — if the LLM call fails, Temporal retries with exponential backoff. No LLM call is ever in the HTTP path. Each agent step is a separate activity, so partial progress is checkpointed automatically.

  • 07 Grievance Resolution Agent DPDP Act compliance requirement. 48h SLA at scale requires automation. Legal risk if skipped.

Advanced AI · External APIs
  • 01 Document Verification Agent Highest trust signal for employees. Depends on TRACES API access (requires MoF partnership) and EPFO scraping stability.
  • 04 Payroll Anomaly Agent High value for CISO/CFO but requires 6+ months of salary insight history to be statistically meaningful. Must be built on mature data.
