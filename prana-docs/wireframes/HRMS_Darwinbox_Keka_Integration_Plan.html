Executive Summary

PRANA's ingest layer today accepts documents via a direct REST API authenticated with HMAC-signed API keys. To reach the enterprise HRMS market, we need plug-and-play connectors for Darwinbox and Keka — the two dominant Indian HRMS platforms covering roughly 60% of the mid-market and enterprise segment.

Each connector is a thin adapter that translates the HRMS's native event format into PRANA's canonical DOC_INGESTED Kafka event, without modifying the core ingest pipeline. The HTTP handler contract remains unchanged: validate → S3 put → 1 DB write → 1 kafka.publish → return 202.

🔸 Darwinbox

  • Auth: API key per tenant (developer portal)
  • Push model: webhooks on document events
  • Pull model: polling employee + document APIs
  • Priority doc types: payslips, offer letters, appraisal letters
  • Typical payload: JSON with binary attachment URL

🔸 Keka

  • Auth: OAuth 2.0 client credentials flow
  • Push model: webhooks (payroll finalized, doc upload)
  • Pull model: REST endpoints with cursor pagination
  • Rate limit: 1,000 req/hour per tenant
  • Bulk export: async job pattern with polling
ℹ Scope: This document covers the connector layer only. The PRANA ingest pipeline (Stage 01–06), Kafka consumers, and Temporal workflows are unchanged. Connectors are a new prana-api/connectors/ module.

Key Design Principles

Adapter Pattern

  • Each connector translates HRMS-native fields to PRANA's canonical IngestPayload
  • Ingest handler remains HRMS-agnostic
  • New HRMS = new connector file only

Idempotency

  • HRMS document ID mapped to PRANA external_ref_id
  • Duplicate webhook = 200 with existing status (not 409)
  • Idempotency key in DB: UNIQUE(tenant_id, external_ref_id)

Webhook Security

  • Darwinbox: HMAC-SHA256 signature on payload
  • Keka: shared webhook secret in header
  • Reject all unverified webhooks with 401

Credential Isolation

  • Each tenant has own HRMS credentials row
  • Credentials encrypted at rest (AES-256-GCM)
  • OAuth tokens cached in Redis with TTL

HRMSConnector Base Class Design

All HRMS connectors extend a common abstract base that enforces the canonical interface. Business logic stays in the connector subclass; the base class owns credential loading, request validation, and the final handoff to the ingest handler.

# prana-api/connectors/base.py
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional
import httpx, logging

logger = logging.getLogger(__name__)

@dataclass
class IngestPayload:
    """Canonical document payload — HRMS-agnostic."""
    tenant_id: str                   # from hrms_connector.tenant_id (NEVER from request)
    employee_id: str                  # PRANA employee_master.employee_id
    external_employee_id: str         # HRMS-native employee identifier
    external_ref_id: str              # HRMS-native document identifier (idempotency key)
    document_type: str                # SALARY_SLIP | FORM_16 | OFFER_LETTER | APPRAISAL | OTHER
    document_period: str              # ISO: "2024-04" for monthly, "2024" for annual
    s3_source_url: str                # Pre-signed URL or direct S3 key (connector fetches bytes)
    original_filename: str
    mime_type: str                    # application/pdf | image/jpeg | etc.
    hrms_source: str                  # "darwinbox" | "keka"
    metadata: dict                    # HRMS-specific passthrough (for audit)

class HRMSConnector(ABC):
    """Abstract base. All connectors must implement these four methods."""

    def __init__(self, tenant_id: str, db_pool, redis, s3_client):
        self.tenant_id = tenant_id
        self.db = db_pool
        self.redis = redis
        self.s3 = s3_client

    @abstractmethod
    async def verify_webhook_signature(self, headers: dict, raw_body: bytes) -> bool:
        """Return True iff webhook signature is valid. Raise 401 if not."""
        ...

    @abstractmethod
    async def parse_webhook_event(self, event_type: str, payload: dict) -> Optional[IngestPayload]:
        """Translate HRMS webhook payload to IngestPayload.
        Return None for event types we don't handle (e.g. employee status updates)."""
        ...

    @abstractmethod
    async def fetch_document_bytes(self, payload: IngestPayload) -> bytes:
        """Download document bytes from HRMS. Called after parse_webhook_event."""
        ...

    @abstractmethod
    async def refresh_credentials(self) -> None:
        """Refresh OAuth token or re-validate API key. Called before each API request."""
        ...

    async def load_connector_config(self) -> dict:
        """Load encrypted HRMS credentials from DB hrms_connector table."""
        row = await self.db.fetchrow(
            """SELECT config_enc, hrms_type, is_active
               FROM hrms_connector
               WHERE tenant_id = $1 AND hrms_type = $2 AND is_deleted = FALSE""",
            self.tenant_id, self.__class__.__name__.lower().replace("connector","")
        )
        if not row or not row["is_active"]:
            raise ValueError(f"No active connector for tenant {self.tenant_id}")
        # Decrypt via KMS envelope — implementation in crypto.py
        return decrypt_connector_config(row["config_enc"])

    async def check_duplicate(self, external_ref_id: str) -> Optional[str]:
        """Return existing document_id if already ingested, else None."""
        row = await self.db.fetchrow(
            """SELECT document_id FROM document
               WHERE tenant_id = $1 AND external_ref_id = $2 AND is_deleted = FALSE""",
            self.tenant_id, external_ref_id
        )
        return str(row["document_id"]) if row else None

Darwinbox Connector

Authentication Flow

Darwinbox uses API key authentication. Each tenant registers PRANA as an integration partner in their Darwinbox developer portal and receives an API key + secret.

# prana-api/connectors/darwinbox.py

DARWINBOX_API_BASE = "https://api.darwinbox.io/v2"  # from platform_config, not hardcoded

class DarwinboxConnector(HRMSConnector):
    HRMS_TYPE = "darwinbox"

    async def refresh_credentials(self):
        config = await self.load_connector_config()
        self._api_key = config["api_key"]        # used in X-API-KEY header
        self._api_secret = config["api_secret"]  # used for HMAC signature verification

    async def verify_webhook_signature(self, headers: dict, raw_body: bytes) -> bool:
        """Darwinbox signs payload with HMAC-SHA256(raw_body, api_secret)."""
        import hmac, hashlib
        received_sig = headers.get("x-darwinbox-signature", "")
        expected_sig = hmac.new(
            self._api_secret.encode(), raw_body, hashlib.sha256
        ).hexdigest()
        return hmac.compare_digest(received_sig, expected_sig)

    async def _darwinbox_get(self, path: str, params: dict = None) -> dict:
        """Authenticated GET to Darwinbox API. Retries once on 401."""
        async with httpx.AsyncClient(timeout=httpx.Timeout(connect=5, read=30)) as client:
            resp = await client.get(
                f"{DARWINBOX_API_BASE}{path}",
                headers={"X-API-KEY": self._api_key, "Accept": "application/json"},
                params=params or {},
            )
            resp.raise_for_status()
            return resp.json()

# Rate limit enforcement using Redis token bucket
async def acquire_rate_limit_slot(redis, tenant_id: str, operation: str, limit: int, window_sec: int):
    key = f"rl:darwinbox:{tenant_id}:{operation}"
    count = await redis.incr(key)
    if count == 1:
        await redis.expire(key, window_sec)
    if count > limit:
        raise RateLimitExceeded(f"Darwinbox {operation} rate limit hit for tenant {tenant_id}")

Edge Cases & Special Handling

⚠ Password-protected payslips: Darwinbox may send is_password_protected: true. Store document as-is, set document.requires_password = TRUE. Do NOT attempt to decrypt. Employee provides password in mobile app session (10-min TTL, in-memory only).
⚠ Employee identity resolution: Darwinbox employee_id may not match PRANA employee_master.employee_id. Resolution ladder: pan_token match → uan match → name+doj fuzzy → create new employee_master row. WorkflowConsumer starts IdentityResolutionWorkflow.
ℹ Bulk historical sync: On initial tenant onboarding, Darwinbox provides a CSV export. Use DarwinboxBulkSyncWorkflow (Temporal, prana-ingest queue) to process. Not a webhook flow.

Keka Connector

OAuth 2.0 Authentication Flow

Keka uses OAuth 2.0 client credentials grant. Each PRANA tenant that uses Keka registers as an OAuth client in their Keka account. PRANA exchanges client credentials for a bearer token (expires in 1 hour).

# prana-api/connectors/keka.py

KEKA_API_BASE = "https://{{company}}.keka.com/api/v1"  # company slug from connector config
KEKA_AUTH_URL = "https://{{company}}.keka.com/connect/token"

class KekaConnector(HRMSConnector):
    HRMS_TYPE = "keka"

    async def refresh_credentials(self):
        config = await self.load_connector_config()
        self._client_id = config["client_id"]
        self._client_secret = config["client_secret"]
        self._company_slug = config["company_slug"]
        self._webhook_secret = config["webhook_secret"]

        # Check Redis cache first (key = keka:token:{tenant_id})
        cached = await self.redis.get(f"keka:token:{self.tenant_id}")
        if cached:
            self._access_token = cached.decode()
            return

        # Token exchange
        async with httpx.AsyncClient(timeout=httpx.Timeout(connect=5, read=15)) as client:
            resp = await client.post(
                KEKA_AUTH_URL.replace("{{company}}", self._company_slug),
                data={
                    "grant_type": "client_credentials",
                    "client_id": self._client_id,
                    "client_secret": self._client_secret,
                    "scope": "keka:payroll:read keka:employees:read keka:documents:read",
                },
                headers={"Content-Type": "application/x-www-form-urlencoded"},
            )
            resp.raise_for_status()
            token_data = resp.json()
            self._access_token = token_data["access_token"]
            expires_in = token_data.get("expires_in", 3600)

            # Cache with 5-min buffer before expiry
            await self.redis.setex(
                f"keka:token:{self.tenant_id}",
                expires_in - 300,
                self._access_token,
            )

    async def verify_webhook_signature(self, headers: dict, raw_body: bytes) -> bool:
        """Keka uses X-Keka-Webhook-Secret header with shared secret."""
        received = headers.get("x-keka-webhook-secret", "")
        return hmac.compare_digest(received, self._webhook_secret)

    async def _keka_get(self, path: str, params: dict = None) -> dict:
        await self.refresh_credentials()
        base = KEKA_API_BASE.replace("{{company}}", self._company_slug)
        async with httpx.AsyncClient(timeout=httpx.Timeout(connect=5, read=30)) as client:
            resp = await client.get(
                f"{base}{path}",
                headers={"Authorization": f"Bearer {self._access_token}"},
                params=params or {},
            )
            if resp.status_code == 401:
                # Token expired early — clear cache and retry once
                await self.redis.delete(f"keka:token:{self.tenant_id}")
                await self.refresh_credentials()
                resp = await client.get(f"{base}{path}", ...)
            resp.raise_for_status()
            return resp.json()

Keka Rate Limit Management

Keka enforces a hard limit of 1,000 requests/hour per OAuth client. On payroll.finalized webhooks, we must fetch payslips for all employees — potentially hundreds of API calls.

# Bulk payslip fetch strategy for payroll.finalized events
async def handle_payroll_finalized(self, payload: dict) -> list[IngestPayload]:
    """
    Strategy: Don't fetch individually per employee.
    Use Keka's bulk export endpoint, then enqueue via Temporal.
    """
    period = derive_period(payload["payPeriod"])  # "2024-04"
    employee_count = payload.get("totalEmployees", 0)

    if employee_count <= 50:
        # Small org — fetch directly with rate limiting
        return await self._fetch_all_payslips_direct(period)
    else:
        # Large org — trigger async bulk export job
        job_id = await self._start_bulk_export_job(period)
        # Temporal workflow polls for completion
        await temporal.start_workflow(
            "KekaBulkExportWorkflow",
            {"tenant_id": self.tenant_id, "job_id": job_id, "period": period},
            id=f"keka-bulk-{self.tenant_id}-{period}",
            task_queue="prana-ingest",
        )
        return []  # Temporal workflow handles ingestion async

async def _fetch_all_payslips_direct(self, period: str) -> list[IngestPayload]:
    """Rate-limited sequential fetch with Redis token bucket."""
    employees = await self._get_all_employees()
    results = []
    for emp in employees:
        await acquire_rate_limit_slot(self.redis, self.tenant_id, "keka_api", 900, 3600)
        doc = await self._keka_get(f"/payroll/payslips", params={"period": period, "employeeId": emp["id"]})
        results.append(self._map_payslip(emp, doc, period))
        await asyncio.sleep(0.1)  # 10 req/sec max to avoid burst
    return results

OAuth Token Refresh Edge Cases

⚠ Token race condition: Multiple concurrent webhook requests may all try to refresh an expired token simultaneously. Use Redis SET NX lock on keka:refresh_lock:{tenant_id} with 10s TTL. Only one request refreshes; others wait on the cached value.
ℹ Token cache key: keka:token:{tenant_id} — stored in Redis under the identity namespace. TTL = expires_in - 300 seconds. Never store the client_secret in Redis — only the derived access token.

    created_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

Error Handling & Reliability

Idempotency

Both connectors guarantee idempotency at two levels:

Level 1 — Webhook dedup

  • hrms_connector_event.hrms_event_id UNIQUE constraint
  • Duplicate webhook → ON CONFLICT DO NOTHING → return 200 with original document_id
  • Log duplicate in connector event table with status IGNORED

Level 2 — Document dedup

  • document.external_ref_id UNIQUE per tenant
  • Re-push of same payslip → return existing document_id
  • Matches HRMS retry-on-failure behavior (Darwinbox retries 3x)

Circuit Breaker Pattern

# prana-api/connectors/circuit_breaker.py
class ConnectorCircuitBreaker:
    """
    States: CLOSED (normal) → OPEN (failing) → HALF_OPEN (testing)
    Stored in Redis: cb:{hrms_type}:{tenant_id}
    """
    FAILURE_THRESHOLD = 5         # open after 5 consecutive failures
    RECOVERY_TIMEOUT = 60         # seconds in OPEN state before HALF_OPEN
    SUCCESS_THRESHOLD = 2         # successes in HALF_OPEN to close

    async def call(self, fn, *args, **kwargs):
        state = await self._get_state()
        if state == "OPEN":
            if await self._recovery_window_passed():
                await self._set_state("HALF_OPEN")
            else:
                raise CircuitOpenError("Circuit OPEN — HRMS API calls suspended")
        try:
            result = await fn(*args, **kwargs)
            await self._on_success()
            return result
        except (httpx.HTTPError, asyncio.TimeoutError) as e:
            await self._on_failure()
            raise

Failed Document Handling

# When document fetch or ingest fails, don't lose it
async def handle_ingest_failure(
    self, event_log_id: str, error: Exception, payload: dict
):
    # 1. Update connector event status
    await self.db.execute(
        """UPDATE hrms_connector_event
           SET processing_status = 'FAILED', error_message = $1
           WHERE event_log_id = $2""",
        str(error)[:500], event_log_id
    )
    # 2. Publish failure event → AuditConsumer → audit_event row
    await kafka_producer.publish("prana.audit.events", {
        "event_type": "CONNECTOR_INGEST_FAILED",
        "tenant_id": self.tenant_id,
        "hrms_type": self.HRMS_TYPE,
        "event_log_id": event_log_id,
        "error": str(error)[:200],
    }, key=self.tenant_id)
    # 3. Publish to notifications → OA-Operator gets portal bell + email
    await kafka_producer.publish("prana.notifications", {
        "notification_type": "CONNECTOR_FAILURE",
        "tenant_id": self.tenant_id,
        "recipient_role": "OA_OPERATOR",
        "event_log_id": event_log_id,
    }, key=self.tenant_id)

Open Questions & Risks

#Risk / QuestionSeverityOwnerResolution path
1 PAN in webhook: Darwinbox and Keka both include PAN in payslip webhooks. If our endpoint has any bug that logs raw_body, PAN leaks to logs. HIGH Security Audit all logging in connector layer. Never log raw_body. Structured logging only with explicit field allowlist. Add SEC-05 rule to enforce_rules.py.
2 Employee identity resolution failure: Darwinbox employee_id won't match existing PRANA employee_master rows for tenants who were already using the direct API before connector. HIGH Backend Build resolution ladder: pan_token → uan → name+doj → auto-create. Manual resolution UI for OA-Admin for failed matches. Block ingestion for unresolvable identities (not silent drop).
3 Keka API instability: Keka's API has documented maintenance windows (Sunday 2–4 AM IST). Webhooks may be held and bulk-delivered after maintenance. MED Backend Idempotency handles duplicate delivery. Circuit breaker handles bulk burst. Temporal DLQ handles partial failures. No action needed during maintenance window — auto-recovers.
4 Darwinbox field schema instability: Darwinbox's webhook payload schema is not versioned and has changed in past (May 2023, Nov 2023). Clients broke silently. MED Backend Pydantic model with extra="allow" for forward compatibility. Required fields validated strictly. Slack alert on unexpected missing required field. Weekly schema diff check in CI.
5 Keka OAuth scope: Unclear if keka:payroll:read scope includes document download or if a separate keka:documents:read scope is needed. Keka docs are inconsistent. MED Integration Contact Keka developer support before implementation. Test in Keka sandbox with both scopes. If separate scope needed, add to OAuth request and connector config.
6 Multi-HRMS per tenant: A tenant may have both Darwinbox and Keka (e.g., switching HRMS mid-year). Currently UNIQUE(tenant_id, hrms_type) prevents this. LOW Product Schema allows multiple connectors per type via removing UNIQUE constraint (additive migration). Decision: allow parallel connectors or require one active at a time? Defer to Product.
7 Bulk historical data: First-time tenant onboarding may require importing 3–5 years of historical payslips (1,000+ documents). Webhook flow not designed for this. LOW Backend Build separate BulkHistoricalSyncWorkflow (Temporal) for onboarding. Accept CSV/ZIP from OA-Admin portal. Out of scope for Phase 1 — schedule for Phase 2.
8 Webhook URL registration: Darwinbox and Keka require tenants to manually register the PRANA webhook URL in their HRMS admin. If URL changes (e.g. Cloudflare Tunnel → ALB), all tenants break. LOW DevOps Publish stable webhook URL early (https://ingest.prana.in/v1/connectors/{hrms}/webhook). Never change this URL. Route internally as needed. Document in partner onboarding guide.
