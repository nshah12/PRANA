Encryption Architecture
11 / 16
🔐 Data Protection Model

Envelope encryption.
Per-employee. Per-tenant key hierarchy.

Three independent encryption layers protect every piece of employee data. A breach at any one layer does not expose data protected by the others.

🔢
PAN Token — Cross-Tenant Deduplication Key
HMAC-SHA256(PAN, platform_secret) · 2ms lifetime · plaintext destroyed after hash
🔒
enc_pan — Format-Preserving Encryption per Employee
FF3-1 FPE(PAN, employee_DEK) · reversible · stored encrypted · never plaintext in DB
🗝
enc_dek — AWS KMS Envelope Encryption
KMS_Encrypt(DEK, tenant_KEK) · customer-managed CMK · ap-south-1 · per-employee DEK
📄
Document Bytes — AES-256 + Application Layer
S3 SSE at rest · per-employee DEK app-layer encryption · in-memory decryption only
🔑
Passwords Argon2id · TOTP AES-256-GCM
Argon2id: time=2, memory=65536, parallelism=2 · TOTP secret encrypted separately
💥 Blast Radius Isolation
Each employee has their own DEK. A DEK compromise exposes exactly one person's documents — not the entire tenant. Mathematics limits the damage.
🚫 Zero Plaintext Policy
No plaintext PAN anywhere in the system after the initial 2ms HMAC window. No logs, caches, or API responses contain raw PAN or salary — ever.
🔑 AWS KMS Key Hierarchy
Platform secret and tenant KEKs managed by AWS KMS (ap-south-1, customer-managed). Key rotation and audit policies enforced by KMS — not application code.
🕒 Password-Protected Documents
Employee provides password → in-memory decryption only → session wiped on expiry (10 min). Nothing persisted. Zero residual data after session end.

AI Extraction Pipeline
13 / 16
🤖 Verified at Every Step · Private by Design

6-stage AI pipeline — raw document to verified vault entry

Every employer-pushed document passes through 6 stages. Produces insights and verification — raw salary figures and identity numbers never reach any database.

STAGE 01 · INGEST
📥
Ingest & Validate
HMAC signature verify · HRMS API key auth · Payload schema validation · S3 put · 1 DB write · Kafka publish DOC_INGESTED · 202 return. No business logic in HTTP path.
✅ 202 in <50ms
STAGE 03 · AI EXTRACTION
🧠
LLM Extraction
Qwen2.5-14B-Instruct processes full document text. Structured JSON output — document type, dates, designation, grade, employer. Designed for Indian HR document formats.
🔵 Full doc in → JSON out
STAGE 05 · AI INSIGHTS
💡
Insights Generation
Llama-3.1-8B generates career insight text. Raw ₹ and PAN never written to DB. Privacy filter is mandatory on all LLM output — blocks any financial figure before storage.
🔒 Insights only — no raw ₹
STAGE 02 · OCR
🔍
OCR
Tesseract (primary) → AWS Textract (fallback). Hindi + English. Scanned PDFs and digital PDFs both handled. Structured text extracted for Stage 03.
✅ Bilingual OCR
STAGE 04 · RESOLUTION
🔗
Identity Resolution
Match extracted identity to employee_master via PAN HMAC token. UAN cross-verify via EPFO. Exception queue raised if match fails — SLA-tracked, OA-Admin resolves.
🔵 EPFO cross-verify
STAGE 06 · VAULT
🏦
Route to Vault
Document routed. Ownership transferred structurally — employer access removed permanently. Employee notified via app push, WhatsApp, and email. Vault URL updated.
✅ Employer locked out
🔒 Privacy Contract
Stages 03 & 05: LLM Input = full document data (needed for extraction)  ·  LLM Output = insights only, no raw ₹, no PAN  ·  DB Write = insights + verified flags only. Enforced in code, not policy.
