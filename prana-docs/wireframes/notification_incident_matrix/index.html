Employee notification cascade
1
WhatsApp WABA — if employee opted in AND Meta template is approved
Opt-out or template not approved → skip to next
2
SMS via MSG91 (primary) → Exotel (fallback on 5 consecutive MSG91 failures)
DND registered or network failure → skip to next
3
Email via SES
Bounced or no email on file → skip to next
4
Push notification (Expo mobile) — if app installed and token valid
App uninstalled or token expired → skip to next
5
In-vault bell — always available, seen on next login
Written to notification_log; visible in vault header badge

1
Email via SES (primary — work email always present)
2
Portal bell — always written, regardless of email delivery status
3
SMS — only for P0 and P1 severity incidents
Bounce / suppression rules
SES bounce webhook → SNS → NotifConsumer → notification_log.status = 'BOUNCED' + insert into email_suppression

All future EMAIL notifications check email_suppression before sending (check_suppression=True).
Suppressed emails log status = 'SUPPRESSED' in notification_log — never silently dropped.
P0 Incident — 30 min SLA
Triggers: Anomaly P0, KMS rotation failed, CSAM detected

Auto-create incident row → assign to CISO + PA
Email + SMS dispatched immediately

T+30 min: If not resolved → SMS escalation to PA + CISO
T+2 hr: incident.status = ESCALATED, PA notified again
P1 Incident — 4 hr SLA
Triggers: Anomaly P1, Foreign IP login, Storage 100%, Platform health degraded

Auto-create incident row → assign to CISO (security) or PA (platform)
Email at creation

T+4 hr: If not resolved → email escalation to PA
T+8 hr: incident.status = ESCALATED
P2 Incident — 24 hr SLA
Triggers: Exception SLA breach (4 hr), CISO manually escalates P3

Auto-create incident row → assign to OA-Admin
No SMS — email only at creation

CISO can manually promote P3 anomaly → P2 incident created
