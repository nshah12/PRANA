▾02 pan_token — What It Is, Why It Exists, How It Is Generated

pan_token is the most important field in the entire PRANA data model. It is the cross-tenant, permanent identity anchor for every person in the system. Without understanding pan_token, none of the deduplication logic makes sense. This section explains it completely — what it is, what it is not, how it is generated, and the security properties that make it work.

🪪
Step 0 — What is NIK / PAN?
NIK stands for National Identity Key — PRANA's internal term for the national identifier used as the vault anchor. In India, this is the PAN (Permanent Account Number) issued by the Income Tax Department. Format: 5 letters + 4 digits + 1 letter (e.g. ABCDE1234F).

Every salaried individual in India has a PAN. Employers already collect it for TDS/Form 16. It is unique per person, issued by the government, and never changes. This makes it the ideal deduplication key — when the same person joins multiple companies, their PAN is the one constant that connects all their employments.

PRANA never stores the plaintext PAN anywhere. The moment it arrives — in an OA upload, an API call, or a batch file — it is transformed into pan_token and then discarded. Nothing in the system can reconstruct the original PAN from pan_token. This is by design.
🔑
What pan_token IS and IS NOT
pan_token IS: A deterministic, one-way transformation of the PAN. Given the same PAN and the same platform_secret, it always produces the same output. This is what makes it a deduplication key — same person = same pan_token, always, across every tenant, across all time.

pan_token IS NOT: The PAN itself. Not a hash of the PAN alone. Not reversible. Not guessable without the platform_secret. Not the same as enc_pan (which is a different field — see below).

pan_token ≠ SHA-256(PAN): A plain SHA-256 of the PAN would be dangerous — anyone with a list of valid PAN numbers could precompute all possible hashes and reverse-lookup whose vault they belong to (rainbow table attack). The HMAC construction prevents this because you need the platform_secret to compute it.
▶ GENERATION ALGORITHM
pan_token generation — step by step
Input → plaintext_NIK = "ABCDE1234F" (received from OA upload, API call, or batch file)

Step 1 → Normalise
NIK = plaintext_NIK.strip().upper() → "ABCDE1234F" (remove spaces, force uppercase)
Validate format: r'^[A-Z]{5}[0-9]{4}[A-Z]{1}$' → reject if invalid (goes to exception queue)

Step 2 → Compute pan_token = HMAC-SHA256(NIK, platform_secret)
platform_secret = 32-byte random key, generated once at platform setup
Stored in AWS KMS (ap-south-1). Never in code, config files, or environment variables.
Retrieved via KMS Decrypt API call at ingest time. Cached in memory for 5 minutes max.

Step 3 → Compute enc_pan = FF3-1 FPE Encrypt(NIK, employee_DEK) (different purpose — see below)

Step 4 → Discard plaintext NIK immediately
del plaintext_NIK ← explicitly zeroed from memory
Never written to any log, queue message, or database column

Result → pan_token = "9f2a8b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b7c8d9e0f1a" (64-char hex string — HMAC-SHA256 output)
▾ Python — pan_token generation 
## pan_token generation — Python (AuthService / IngestService)
import hmac, hashlib, re
import boto3  # AWS SDK — for KMS retrieval

## Step 1: Retrieve platform_secret from KMS (cached 5 min)
def get_platform_secret() -> bytes:
    kms = boto3.client('kms', region_name='ap-south-1')
    response = kms.decrypt(
        CiphertextBlob = ENCRYPTED_PLATFORM_SECRET,  # stored in config, encrypted
        KeyId          = PLATFORM_KEK_ARN
    )
    return response['Plaintext']  # 32 bytes

## Step 2: Validate and normalise NIK
def validate_nik(raw_nik: str) -> str:
    nik = raw_nik.strip().upper()
    if not re.match(r'^[A-Z]{5}[0-9]{4}[A-Z]{1}$', nik):
        raise ValueError(f"Invalid NIK format: {raw_nik[:3]}***")  # never log full NIK
    return nik

## Step 3: Compute pan_token
def compute_pan_token(nik: str, platform_secret: bytes) -> str:
    return hmac.new(
        key       = platform_secret,
        msg       = nik.encode('utf-8'),
        digestmod = hashlib.sha256
    ).hexdigest()  # 64-char lowercase hex string

## Step 4: Compute enc_pan using FF3-1 FPE
## (preserves format — encrypted output looks like a PAN)
from ff3 import FF3Cipher

def compute_enc_pan(nik: str, employee_dek: bytes) -> str:
    cipher = FF3Cipher(employee_dek, tweak=b'PRANA001')
    return cipher.encrypt(nik)  # output same length/format as PAN

## Step 5: Full pipeline (called at Stage 02 of ingest)
def process_nik(raw_nik: str, employee_dek: bytes) -> dict:
    platform_secret = get_platform_secret()
    nik             = validate_nik(raw_nik)

    pan_token = compute_pan_token(nik, platform_secret)
    enc_pan   = compute_enc_pan(nik, employee_dek)

    ## Explicitly discard plaintext
    del nik
    del platform_secret

    return {'pan_token': pan_token, 'enc_pan': enc_pan}

## Usage
result = process_nik("ABCDE1234F", employee_dek)
## result = {'pan_token': '9f2a8b3c...', 'enc_pan': 'XYZPQ7891K'}
## Plaintext "ABCDE1234F" is now gone from memory
▶ pan_token vs enc_pan — TWO DIFFERENT FIELDS, TWO DIFFERENT PURPOSES
Property pan_token enc_pan
What it is HMAC-SHA256 of the PAN FF3-1 Format-Preserving Encrypted PAN
Input NIK + platform_secret (global key) NIK + employee_DEK (per-person key)
Output format 64-char hex string — looks nothing like a PAN 10-char alphanumeric — same format as a PAN (e.g. XYZPQ7891K)
Reversible? NO — one-way. Cannot recover PAN from pan_token even with platform_secret. YES — decrypt with employee_DEK recovers original PAN. Used only when PAN needs to be retrieved for verification.
Primary purpose Deduplication key. Same person across all tenants = same pan_token. Used in identity resolution lookup. Reversible storage. For cases where PAN needs to be recovered (e.g. regulatory verification, AI extraction cross-check).
Key dependency platform_secret — one global key, held in KMS. All tenants share this key (which is why same PAN always produces same pan_token). employee_DEK — unique per person. Different for every employee_user. Means enc_pan of the same PAN is different for different people (if two employees have same PAN — impossible in practice, but guarded).
Queryable? YES — indexed. Primary lookup: SELECT * FROM employee_user WHERE pan_token = $1 NO — not indexed. Only decrypted on demand. Never used as a query key.
If compromised Attacker knows two employees are the same person (linkability) but cannot recover PAN. Rotate platform_secret → recompute all pan_tokens. Attacker recovers PAN for that one employee only (needs that employee's DEK). Rotate employee_DEK → re-encrypt enc_pan.
🔐
platform_secret — What It Is and How It Is Managed
What it is: A 32-byte (256-bit) random secret key, generated once at PRANA platform setup using a cryptographically secure random number generator. It never changes (changing it would require recomputing pan_token for every employee in the system — a massive operation).

Where it lives: AWS KMS ap-south-1 as a Customer Managed Key (CMK). The plaintext key never touches disk, code, or config files. IngestService calls kms:Decrypt at ingest time to get it, uses it in memory for the HMAC computation, then discards it. KMS logs every usage — providing a complete audit trail of every pan_token generation event.

Who can access it: Only the IngestService IAM role has kms:Decrypt permission on this key. No human, no OA user, no PA, no other service can access it. Even the PRANA engineering team cannot retrieve the plaintext key from KMS once it is created (KMS HSMs are FIPS 140-2 Level 3 — the key cannot be exported).

Rotation: Intentionally NOT rotated. Rotating platform_secret would produce different pan_token values for the same PAN — breaking all existing identity links across tenants. If the key is ever compromised, the response is: (1) generate new platform_secret, (2) bulk-recompute all pan_tokens offline, (3) swap atomically. This is a break-glass procedure, not a routine rotation.
🛡
Security Properties of the pan_token Design
Pre-image resistance: Given pan_token, you cannot recover the PAN. HMAC-SHA256 is cryptographically one-way.

Rainbow table resistance: Unlike SHA-256(PAN), an attacker with a list of valid PAN numbers cannot precompute pan_tokens — they need the platform_secret, which never leaves KMS.

Collision resistance: Two different PANs will never produce the same pan_token (HMAC-SHA256 collision probability is astronomically small — 2⁻²⁵⁶).

Cross-tenant linkability: The same PAN always produces the same pan_token. This is intentional — it's how deduplication works. The risk (a compromised pan_token reveals that two employers hired the same person) is accepted as a design trade-off and is lower risk than storing PAN directly.

▾04 The Core Insight — Three Distinct Identities

Before reading any table, understand this. Most HR systems conflate three things that PRANA deliberately keeps separate. Getting this wrong causes duplicate records, split vaults, and broken portability — the exact problems PRANA exists to solve.

① The Person
Permanent. Unique. Never changes.

"Rahul Sharma with PAN ABCDE1234F"

Anchored by pan_token = HMAC-SHA256(PAN, platform_secret). Same person across every employer, forever. Stored in: employee_user
② The Vault User
Permanent. Created once. Never duplicated.

"Rahul Sharma's PRANA account"

Login credentials, TOTP, session management. One record, exists independent of any employer. Stored in: employee_user (same table — person IS the vault user)
③ The Employment
Temporary. One per job. Created fresh each time.

"Rahul at NPCI 2019–2024"
"Rahul at ABCD Bank 2024–now"

Carries designation, dept, reporting manager. Stored in: employee_master
⚠
The Duplication Trap — How Most Systems Fail
When Rahul leaves NPCI and joins ABCD Bank, ABCD Bank's HR uploads him. A naive system creates a new "employee" record — now Rahul exists twice. His NPCI documents are in one record, ABCD Bank documents in another. His vault is split. He can't see his full history. Portability is broken.

PRANA's fix: pan_token is computed from the PAN. Same PAN = same pan_token = same employee_user. ABCD Bank upload finds the existing employee_user and creates a new employee_master row pointing to it. One vault. Complete history. Zero duplication.

▶ SCENARIO 1: Rahul leaves NPCI and joins ABCD Bank
Step-by-step deduplication flow
Step 1 → ABCD Bank HR uploads Rahul
Input: { PAN: "ABCDE1234F", name: "Rahul Sharma", doj: "2024-07-01", designation: "VP Payments" }

Step 2 → Stage 02: Encryption Boundary fires immediately
pan_token = HMAC-SHA256("ABCDE1234F", platform_secret)
→ pan_token = "9f2a8b3c..." (same value as when NPCI uploaded him in 2019)
Plaintext PAN discarded immediately. Never stored anywhere.

Step 3 → Identity Resolution Ladder 1
SELECT employee_user_id FROM employee_user WHERE pan_token = '9f2a8b3c...'
→ FOUND: employee_user_id = XYZ (Rahul's permanent identity, created in 2019)

Step 4 → Check for existing active employment at ABCD Bank
SELECT * FROM employee_master WHERE employee_user_id = XYZ AND tenant_id = ABCD_BANK AND dol IS NULL
→ NOT FOUND — no current employment at ABCD Bank. Safe to create.

Step 5 → Create NEW employee_master row — not a new employee_user
INSERT INTO employee_master (
employee_uuid = NEW UUID (BBB), ← new employment ID
employee_user_id = XYZ, ← links to EXISTING person
tenant_id = ABCD_BANK_UUID,
doj = '2024-07-01',
designation = 'VP Payments'
)

Step 6 → Log career_event
event_type = 'JOINED', event_title = 'Joined ABCD Bank as VP Payments'

RESULT →
employee_user XYZ — one record, same vault, same enc_dek (documents stay encrypted under same key)
employee_master AAA — NPCI employment (dol = 2024-06-30, can_push = FALSE after window)
employee_master BBB — ABCD Bank employment (dol = NULL, active)
✓ Rahul's vault shows ALL documents from BOTH organisations. Zero duplication.

▾18 Automatic Role Deactivation on Exit

When a roled user (OA-Operator, OA-Admin, CHRO, CFO, Tenant CISO) leaves the organisation, their employee_master.dol is set — the same field used for any employee's exit. If their oa_user account is linked via linked_employee_user_id, this single field-set can automatically cascade to deactivate their organisational role — while leaving their personal vault (employee_user) permanently untouched.

🎯
The trigger: dol arriving, not dol being SET
dol is often set in advance — HR records a resignation with a notice period (e.g. Vikram resigns on 1 Aug, dol = 31 Aug, but retains admin access through the notice period to do his job). Deactivating immediately when dol is set would lock him out during notice — wrong.

The correct trigger is dol arriving — today's date reaching the recorded leaving date. This is a durable, scheduled action — exactly what Temporal workflows with timers are for.
▶ THE CASCADE — EmployeeExitWorkflow (Temporal)
Triggered when employee_master.dol is SET — fires when dol ARRIVES

▾19 Time-Bound Policy Locks — Lock, Wait, Auto-Unlock or Admin Override

When an anomaly rule fires — e.g. BULK_ACCESS_ANOMALY on document_access_log — the system locks the account for a fixed duration (e.g. 24 hours). If no admin acts, it auto-unlocks when the duration elapses. If an admin investigates and clears it early, the lock ends immediately and the timer is cleanly cancelled — with a single, immutable audit trail either way.

⚠
The race condition this design must handle
Vikram is locked for 24 hours at 10:00 (POLICY_LOCK, scheduled_unlock_at = next day 10:00). At 10:02, an OA-Admin investigates, confirms it was a false positive, and clicks "Unlock now."

Without careful design: the 24-hour timer is still running. At 10:00 the next day it fires and either (a) does nothing useful — wasted — or worse (b) writes a conflicting AUTO_UNLOCK event on top of the admin's ADMIN_ENABLED event, creating two "this account was unlocked" rows for one actual unlock — breaking the clean 1:1 reversal link in reversed_by_event_id.

The fix: the timer's completion handler checks reversed_by_event_id IS NULL on the original lock event before writing anything. If the admin already resolved it, the handler is a no-op. This single idempotency check is what makes the design correct regardless of which engine (Temporal signal, cron poll, Step Functions) drives the timer.

class AccountLockService:

    async def apply_policy_lock(self, user_type, user_id, reason_code, duration_hours):
        event_id = await insert_account_status_event(
            user_type   = user_type, user_id = user_id,
            event_type  = 'POLICY_LOCK',
            from_status = 'ACTIVE', to_status = 'LOCKED',
            reason_code = reason_code,
            scheduled_unlock_at = now() + timedelta(hours=duration_hours),
            actor_type  = 'system'
        )
        await update_status(user_type, user_id, 'LOCKED')
        return event_id

    async def auto_unlock(self, user_type, user_id, lock_event_id):
        ## ── Idempotency check — THIS is what makes Path A/B race-safe ──
        lock_event = await get_account_status_event(lock_event_id)
        if lock_event.reversed_by_event_id is not None:
            return  # admin already resolved it — no-op

        new_event_id = await insert_account_status_event(
            user_type   = user_type, user_id = user_id,
            event_type  = 'AUTO_UNLOCK',
            from_status = 'LOCKED', to_status = 'ACTIVE',
            reason_code = 'COOLDOWN_EXPIRED',
            actor_type  = 'system'
        )
        await link_reversal(lock_event_id, new_event_id)
        await update_status(user_type, user_id, 'ACTIVE')

    async def admin_unlock(self, user_type, user_id, lock_event_id, admin_id, reason_note):
        new_event_id = await insert_account_status_event(
            user_type   = user_type, user_id = user_id,
            event_type  = 'ADMIN_ENABLED',
            from_status = 'LOCKED', to_status = 'ACTIVE',
            reason_code = 'ADMIN_REQUEST', reason_note = reason_note,
            actor_type  = 'oa_admin', actor_id = admin_id
        )
        await link_reversal(lock_event_id, new_event_id)
        await update_status(user_type, user_id, 'ACTIVE')
        return new_event_id

@workflow.defn
class PolicyLockWorkflow:
    resolved: bool = False

    @workflow.run
    async def run(self, user_type, user_id, reason_code, duration_hours):
        lock_event_id = await workflow.execute_activity(
            AccountLockService.apply_policy_lock,
            args=[user_type, user_id, reason_code, duration_hours]
        )
        try:
            await workflow.wait_condition(
                lambda: self.resolved,
                timeout=timedelta(hours=duration_hours)
            )
            ## Path A — signal arrived. admin_unlock already wrote everything.
        except TimeoutError:
            ## Path B — 24h elapsed with no admin action.
            await workflow.execute_activity(
                AccountLockService.auto_unlock,
                args=[user_type, user_id, lock_event_id]
            )

    @workflow.signal
    def resolve(self):
        self.resolved = True

▾21 Engine Independence — Why Temporal Is Replaceable

Temporal (Apache 2.0, self-hostable, free forever as software) is PRANA's chosen workflow engine. But "Temporal becomes paid" isn't really the risk — Temporal Cloud (the managed service) pricing could change, or the team could decide for other reasons to move to AWS Step Functions, Airflow, or a cron+queue fallback. This section defines the design rule that keeps that door open — and is good practice regardless.

▶ WHERE TEMPORAL "LEAKS" INTO YOUR APPLICATION IF YOU'RE NOT CAREFUL
TIGHT COUPLING (hard to remove)
Business logic written INSIDE workflow functions — e.g. the last-OA-Admin check, account_status_event writes, ONLY exist as Temporal "activities"
Temporal-specific APIs (workflow.sleep_until, signal_workflow) called directly from API handlers / services anywhere outside one boundary layer
temporal_workflow_id stored as a foreign key elsewhere in the schema
LOOSE COUPLING (Temporal is replaceable)
Temporal is ONLY a scheduler/timer + retry mechanism
All business logic lives in plain service classes (AccountLockService etc.) that Temporal CALLS — a cron job or queue consumer could call the same classes
The application database (employee_master, account_status_event, oa_user) is the source of truth — NOT Temporal's internal state
A single thin adapter layer translates "schedule X to happen at time T" into Temporal calls — swap the adapter, not the business logic
🔗
What genuinely ties you to Temporal (and is worth it — but is the actual lock-in)
1. Workflow history/visibility UI — Temporal's killer feature for ops/debugging. Step Functions has a similar console; cron has none.

2. Signal-based cancellation — clean, immediate. Cron-based requires the idempotency check pattern (reversed_by_event_id IS NULL) shown in PolicyLockWorkflow above — slightly less elegant but works fine.

3. Complex multi-step sagas — the EmployeeExitWorkflow 7-step cascade is readable as sequential Python. On Step Functions it's a state machine JSON. On cron, it's a series of independently-idempotent steps.

None of these are data-model lock-in — they're orchestration-code lock-in, confined to the adapter layer.

▾26 login_attempt_log

Every login attempt — successful or failed — across all user types, both authentication factors, and every entry point. This is the dedicated login audit table. It answers what no other table can: the complete per-user login history with consecutive failure counts, device fingerprints, and IP anomaly flags.

⚠
Why not just use audit_event or user_session?
user_session only records successful logins — a session is created only after both factors pass. Failed attempts leave no trace there.

audit_event is the immutable ledger — it records LOGIN_SUCCESS and LOGIN_FAILED as events, but is not optimised for per-user login history queries or consecutive failure tracking.

login_attempt_log is the dedicated, queryable table for: per-user history, consecutive failure counts, device/IP forensics, and brute-force detection. It covers both PASSWORD and TOTP steps separately — so you can see exactly where in the flow an attacker stopped.
🔒
identifier_entered is stored as SHA-256 hash — never plaintext
When a login fails and we don't know which user it was (unknown email/mobile), we store SHA-256(identifier_entered) — not the raw email or mobile number. This prevents the login log from becoming a directory of email addresses. We can still answer "was this specific identifier used in failed attempts?" by computing SHA-256 and querying — without exposing the plaintext in the DB.
login_attempt_log
Every login attempt — both factors, all user types, success and failure — with full forensic context
SECURITY APPEND-ONLY ALL USER TYPES
Coverage: Three entry points × two factors × all outcomes. Employee at prana.in/login → PASSWORD attempt → TOTP attempt → both logged separately. OA user at prana.in/org/login → same. Portal Admin at prana.in/admin/login → same. A failed PASSWORD attempt and a failed TOTP attempt are separate rows — you can see exactly which factor an attacker failed on and how many times.

Links to other tables: On success (outcome=SUCCESS, attempt_type=TOTP), session_id is populated linking to user_session. On 5th consecutive failure, the lockout that follows writes to account_status_event referencing the failure count and IP from this table.

▾27 Login Forensic Capture — Two-Phase Enrichment, trusted_device, device_credential

login_attempt_log captures the maximum forensically-useful data at every login — but a third-party IP-intelligence lookup (VPN/proxy/ASN/geo) takes 100–300ms and must never block or slow down the login itself. This section defines the two-phase capture model, the trusted_device registry that turns device fingerprints into a real anomaly signal, and device_credential — the table that makes app-based biometric MFA (§29) possible.

▶ TWO-PHASE CAPTURE — SYNC AT LOGIN, ASYNC ENRICHMENT AFTER
Phase 1 — SYNCHRONOUS (blocks login response, target <50ms)
Client sends with login request:
device_fingerprint_hash — SHA-256 of canvas/WebGL/fonts/screen/timezone composite, computed client-side
request_id — distributed trace ID, generated by client or API gateway
client_reported_tz, client_reported_time — for display reconciliation only
entry_point — DIRECT / EMAIL_LINK / QR_CODE / APP_DEEPLINK / SHARE_LINK_REDIRECT
Server, same transaction:
ip_address (XFF-validated), user_agent, device_type — from request headers
SELECT FROM trusted_device WHERE device_fingerprint_hash = $1 AND user_id = $2
→ is_known_device = (row found)
INSERT INTO login_attempt_log (... enrichment_status = 'PENDING')
→ Login proceeds immediately. User experience unaffected.

Phase 2 — ASYNCHRONOUS (fire-and-forget, completes within seconds)
Background worker picks up PENDING rows (idx_lal_enrichment_pending)
Call IP intelligence provider (IPQualityScore / MaxMind GeoIP2 Anonymizer) with ip_address
UPDATE login_attempt_log SET is_vpn_or_proxy=.., is_tor=.., is_datacenter_ip=..,
asn=.., isp_name=.., ip_risk_score=.., geo_confidence_km=.., geo_lat=.., geo_lon=..,
enrichment_status='COMPLETE', enriched_at=NOW()
WHERE attempt_id = $1
If enrichment reveals severe risk →
PolicyLockWorkflow triggered AFTER THE FACT (§19) — login already succeeded,
but account is now locked pending review. Slightly delayed, but never degrades
login UX for the 99.99% of legitimate attempts.
⚖
What this buys you — and what it doesn't
This maximizes forensic reconstruction capability — when a dispute arises ("I never logged in from Lagos"), PRANA can produce: exact IP + ASN ("Vodafone Nigeria mobile, not a VPN"), device fingerprint ("never seen on this account before"), geolocation with confidence radius, and the full session's subsequent actions in one query.

It does not make the system "full proof." A sufficiently resourced attacker can spoof user-agent, route through residential proxies (is_vpn_or_proxy=false even when malicious), or hijack an already-authenticated session (device fingerprint doesn't help here). The honest claim is "maximizes investigation speed and conclusiveness" — not "prevents all attacks." Overclaiming here creates liability if ever tested in an audit or legal proceeding.

device_credential

Why this is structurally different from trusted_device: trusted_device is a heuristic fingerprint — useful for anomaly detection, but not cryptographically unforgeable. device_credential is a real public/private keypair generated and held in the device's Secure Enclave (iOS) / StrongBox (Android) — the private key never leaves the hardware. Biometric unlock (Face ID / fingerprint) gates access to this key, which then signs a login challenge. The server verifies the signature against the stored public_key. This is the same principle as WebAuthn/FIDO2, delivered via the PRANA app instead of a browser (browsers cannot access platform biometrics for server-side verification — only as a local unlock, which is what this table formalizes for the app).

This table is employee-only — the PRANA mobile app (§29) is scoped exclusively to the employee role.

▾28 Self-Upload DLP Policy — Four-Layer Model

Employees can upload documents to their personal vault via the "Upload Document" button. The risk this raises for enterprise customers is not document fraud — it is data exfiltration: a confidential corporate document, uploaded once into an employee's permanent personal vault, persists forever, is outside the company's DLP/MDM controls, and can be C-Shared externally at any point — even years after the employee has left. For BFSI/PSU tenants (NPCI, banks), an unaddressed exfiltration channel is an adoption blocker.

⚖
Why network-based blocking alone is not enough
Hiding the "Upload" button on office IPs is a legitimate deterrent — but it is trivially bypassed (upload from home that evening, or via cellular data while physically in the office). If PRANA's only claim were "we block this on your network," the obvious follow-up — "what stops the same upload from a coffee shop?" — has no good answer. A false sense of security is worse than an honest one. The real protection is Layer 4 (tenant-configurable policy) + Layer 3 (complete audit trail) — "you control the posture, and everything is logged and forensically reconstructable" is a defensible position; "we technically prevent it" is not.
LayerWhat it doesWhat it doesn't do
1
Network		 Hides "Upload Document" when the employee's current session IP matches tenant.office_ip_ranges or login_attempt_log.asn ∈ tenant.office_vpn_asns — and only while employee_master.dol IS NULL. Does not stop upload from elsewhere (home, mobile data, coffee shop). Deterrent/nudge only — complements existing CASB/DLP (Netskope, Zscaler), does not replace it.
2
Content		 Lightweight AI classification at upload time — does the file show corporate markings (letterhead/logo of a different company than the employee's current employer, "Confidential" / "Internal Use Only" / "Proprietary" terms)? If flagged, shows a friction/consent dialog before proceeding. Not full content-inspection DLP. Heuristic, imperfect — false positives on legitimate personal docs that happen to mention "confidential." It is friction + informed consent, not a hard block.
3
Audit		 Every self-upload — regardless of Layer 1/2 outcome — writes a complete forensic record to audit_event and document.self_upload_context: file_hash, IP, was_on_office_network, content_classification_flag, consent_acknowledged, device_fingerprint_hash. Does not prevent anything — enables accountability after the fact. This is what "saves you" in a post-incident investigation: complete reconstruction of when, where, and whether the employee was warned.
4
Policy		 tenant.self_upload_policy — Tenant CISO/OA-Admin chooses: ALLOWED / ALLOWED_WITH_WARNING (default) / BLOCKED_ON_OFFICE_NETWORK / BLOCKED_ENTIRELY. Turns "does PRANA prevent this?" into "the tenant configures their own DLP posture, PRANA enforces it." Requires the OA-Admin to actually configure it — a tenant that leaves the default may have a weaker posture than they assume. Onboarding flow should surface this choice explicitly, not bury it.
▶ LAYER 1 — OFFICE-NETWORK DETECTION (SESSION START)
1. Employee logs in → login_attempt_log row written (current session's ip_address, asn)
2. Server checks: employee_master.dol IS NULL (currently employed)?
NO (alumni) → Layer 1 does not apply — alumni have no "office network" to protect against
YES (active) → continue
3. ip_address <<= ANY(tenant.office_ip_ranges) OR asn = ANY(tenant.office_vpn_asns) ?
YES → "Upload Document" button hidden for this session
NO → button shown (subject to Layer 2 at click-time)
Re-evaluated per session — an employee working from home on Monday and the office on Tuesday sees the button appear/disappear accordingly. Not a permanent flag on the account.

1. Employee selects a file and clicks "Upload"
2. Lightweight classification pass (reuses existing AI extraction pipeline):
Does the document contain a letterhead/logo NOT matching the employee's current tenant?
Does it contain terms: "Confidential", "Internal Use Only", "Proprietary", "Do Not Distribute"?
3. Flagged?
NO → upload proceeds normally, source_type='EMPLOYEE_SELF_UPLOAD', content_classification_flag=NULL

What this buys you, six months later
If a confidential document surfaces externally and is traced back to having been in an employee's PRANA vault, PRANA can produce the complete record: exactly when it was uploaded, from where, whether the AI classifier flagged it as containing corporate markings, and whether the employee explicitly acknowledged a warning before proceeding. 📱
Mobile app — self-upload allowed by default, Layer 1 does not apply
From a phone, an employee cannot normally access company documents (no access to office file shares, internal drives, or printed-and-scanned confidential material via the same casual flow a laptop allows). The PRANA mobile app (§30) therefore allows self-upload by default — Layer 1's office-network/VPN check is a desktop-session concept and does not apply to the app.

This is a calculated, not absent, control: Layer 3's immutable audit trail applies identically regardless of channel — every mobile self-upload still writes file_hash, device_fingerprint_hash (from device_credential, §27), and timestamp to document.self_upload_context and audit_event. If Layer 4 policy is BLOCKED_ENTIRELY, the upload feature is absent from the app too — policy is channel-independent even where Layer 1 is not.
