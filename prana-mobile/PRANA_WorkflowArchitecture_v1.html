Overview & Design Principles

PRANA uses Temporal as its sole mechanism for all async, time-based, and multi-step operations. There are no cron jobs, no Celery workers, no OS-level schedulers, and no polling loops anywhere in the system. Every operation that involves waiting, retry, human input, or multiple steps is a named Temporal workflow defined in this document.

The core rule

If it waits, it is a workflow. If it retries, it is an activity. If it takes more than one step, it is a workflow. This rule has no exceptions in PRANA. The engineer who asks "should I use a cron for this?" has already answered the question.

Why Temporal, not cron

Cron jobs have three failure modes that PRANA cannot tolerate: they miss their window if the pod dies during execution; they have no built-in retry on failure; and they have no durability — if the scheduler crashes, the job is simply lost. For a 7-year document retention clock, a DPDP 72-hour breach notification SLA, or a pan_token re-derivation across 1 crore employees, these failure modes are not acceptable.

Temporal solves all three: workflows replay from their last checkpoint on worker crash, activities retry independently with configurable backoff, and workflow state persists in YugabyteDB with the same active-active guarantees as the rest of the platform data.

Core Patterns

All 53 workflows are built from five composable patterns. Understanding these patterns makes every individual workflow definition below immediately readable.

Pattern 1 — Durable timer

The most common pattern. A workflow starts, records its intent, sleeps until a specific time or duration, then acts. The sleep is durable — it survives pod restarts, deploys, and region failures.

# Pattern: durable timer @workflow.defn class ExampleTimerWorkflow: async def run(self, input: TimerInput) -> None: # Duration read from config at trigger time — not hardcoded duration = timedelta(days=input.duration_days) await workflow.sleep(duration) # Act after timer fires await workflow.execute_activity( perform_action, input.payload, schedule_to_close_timeout=timedelta(minutes=10) )

Pattern 2 — Signal-driven timer (interruptible)

A timer workflow that can be interrupted early by an external signal. Used for: elevation windows (admin ends early), lockout cooldowns (admin unlocks), erasure confirmation windows (employee cancels), share expiry (employee revokes).

@workflow.defn class InterruptibleTimerWorkflow: def __init__(self): self._early_exit = False self._exit_reason = None @workflow.signal async def cancel_early(self, reason: str) -> None: self._early_exit = True self._exit_reason = reason async def run(self, input: Input) -> None: # Wait for either: timer fires OR signal arrives deadline = workflow.now() + timedelta(seconds=input.duration_seconds) await workflow.wait_condition( lambda: self._early_exit, timeout=input.duration_seconds ) if self._early_exit: await workflow.execute_activity(on_early_exit, self._exit_reason, ...) else: await workflow.execute_activity(on_timer_expiry, input.payload, ...)

Pattern 3 — Temporal Schedule (replaces cron)

Recurring workflows use Temporal Schedules — not OS cron. Schedules are created once (at service startup or config change) and run forever. The schedule spec is stored in Temporal itself and is updateable via the Temporal API without redeployment. Cadence is read from platform_config/tenant_config at schedule creation time and updated when config changes.

# Schedule created at service startup (idempotent) async def ensure_schedule(client: Client, config: ConfigService): interval = await config.get_int("platform_summary_interval_minutes") handle = client.get_schedule_handle("platform-summary") try: await handle.describe() # exists — check if interval changed await handle.update(lambda s: s.with_spec( ScheduleSpec(intervals=[ScheduleIntervalSpec( every=timedelta(minutes=interval) )]) )) except RPCError: # does not exist — create await client.create_schedule( "platform-summary", Schedule( action=ScheduleAction(workflow=PlatformSummaryWorkflow), spec=ScheduleSpec(intervals=[ ScheduleIntervalSpec(every=timedelta(minutes=interval)) ]) ) )

Pattern 4 — Continue-As-New (perpetual workflows)

Used for workflows that must run forever without accumulating unbounded history. The workflow periodically restarts itself with fresh state, carrying forward only what it needs. Used by: AnomalyDetectionWorkflow, KMSKeyRotationWorkflow, HMACSecretRotationWorkflow, RetentionWorkflow (at 7-year boundary).

@workflow.defn class PerpetualWorkflow: async def run(self, input: PerpetualInput) -> None: events_processed = input.events_processed_so_far while events_processed < RENEW_THRESHOLD: event = await workflow.execute_activity(consume_next_event, ...) await workflow.execute_activity(process_event, event, ...) events_processed += 1 # Restart with fresh history, carry forward position workflow.continue_as_new(PerpetualInput( events_processed_so_far=0, # reset counter checkpoint=input.checkpoint # carry forward position ))

Pattern 5 — Human signal (multi-actor workflows)

Workflows that require a human decision before proceeding. The workflow waits indefinitely for a signal, with an optional timeout for SLA enforcement. Used by: ElevationWorkflow, StorageExpansionWorkflow, OnboardingReviewSLAWorkflow, TenantMigrationWorkflow.

@workflow.defn class HumanSignalWorkflow: def __init__(self): self._decision = None @workflow.signal async def approve(self, actor_id: str) -> None: self._decision = ("APPROVED", actor_id) @workflow.signal async def reject(self, actor_id: str, reason: str) -> None: self._decision = ("REJECTED", actor_id, reason) async def run(self, input: Input) -> None: await workflow.execute_activity(notify_approver, input, ...) # Wait for human decision, with SLA timeout decision_received = await workflow.wait_condition( lambda: self._decision is not None, timeout=timedelta(days=input.sla_days) ) if not decision_received: # SLA breach await workflow.execute_activity(escalate_sla_breach, input, ...) return decision, actor = self._decision[:2] if decision == "APPROVED": await workflow.execute_activity(on_approved, input, ...)

Configuration Model

No workflow duration or schedule is hardcoded. All configurable values are read from platform_config at workflow trigger time and passed as workflow input. The workflow itself stores the value it received — changing config after a workflow starts does not affect in-progress workflows (intentional — a 30-day erasure window doesn't change mid-erasure).

Resolution order

get_config(key, tenant_id) → checks tenant_config first → falls back to platform_config default. Tenant-level overrides always win. Config changes apply to new workflow instances only — never retroactively.

Engine Independence

All business logic lives in plain Python service classes with zero Temporal imports. Temporal is the orchestration shell — it calls into service methods, not the other way around. This means:

  • Every workflow can be tested without a Temporal cluster
  • Service logic can be called from REST endpoints, CLI scripts, or integration tests directly
  • Migrating to a different orchestrator (Step Functions, Prefect) requires changing only the thin workflow shell — zero business logic rewrite
# services/account_lock.py — zero Temporal imports class AccountLockService: async def apply_lock(self, user_id: str, reason: str, duration_s: int) -> str: # Pure business logic. No workflow engine here. await self.db.update_status(user_id, "LOCKED") event_id = await self.db.write_status_event(user_id, reason) await self.notifier.send_lock_alert(user_id) return event_id # workflows/totp_lockout.py — thin Temporal shell @workflow.defn class TOTPLockoutWorkflow: async def run(self, input: LockoutInput) -> None: lock_svc = AccountLockService() event_id = await workflow.execute_activity( lock_svc.apply_lock, input.user_id, "TOTP_ATTEMPTS_EXCEEDED", input.cooldown_seconds, schedule_to_close_timeout=timedelta(minutes=5) ) # Durable sleep — survives restarts await workflow.sleep(input.cooldown_seconds) await workflow.execute_activity( lock_svc.apply_unlock, input.user_id, event_id, "TOTP_COOLDOWN_EXPIRED", schedule_to_close_timeout=timedelta(minutes=5) )

Sequential activities. One file failure never blocks sibling workflows. Temporal replays from last checkpoint on worker crash.

Activity retry policy
Max 3 attempts per activity. Exponential backoff starting at 1s. Non-retryable: CSAM detection, quota exceeded. LLM activities: max 3 retries, then LOW_CONFIDENCE exception path.

Steps

01
Batch Ingestion activity — Write document row with pipeline_status=QUEUED. Generate staging S3 key. Return document_id.
02
Encryption Boundary activity — OCR if needed (Textract → Tesseract fallback). Regex extract NIK into memory variable (<2ms lifecycle). Compute pan_token = HMAC-SHA256(NIK, platform_secret) + enc_pan = FF3-1(NIK, emp_DEK). Zero plaintext NIK from memory. Replace NIK in text with [NIK_REDACTED]. Return redacted_text + pan_token + enc_pan.
03
Safety Scan activity — ClamAV virus scan. NSFW detection (SafeSearch). CSAM PhotoDNA hash check. If any flag: quarantine S3 move + OA alert + write audit_event SCAN_QUARANTINED. If CSAM: start CSAMReportingWorkflow + set legal_hold=TRUE. CLEAN → proceed.
04
LLM Extraction activity — Schema-specific prompt + redacted_text → Bedrock Claude Sonnet (ap-south-1). Returns JSON with per-field confidence + overall_confidence. overall_confidence < 0.60 → LOW_CONFIDENCE exception. Fields written to document.extracted_fields JSONB.
05
Identity Resolution activity — Run 4-level ladder: pan_token exact → Emp ID exact → Name+DOJ fuzzy → Embedding cosine. Resolved → proceed. Unresolved → write exception_queue row + wait for exception_resolved signal (up to 7 days). Signal carries employee_uuid.
06
Tag & Route activity — Write immutable metadata tag (pan_token, enc_pan, doc_type, period, resolution_method). Move S3 object from staging/ to permanent prefix. Set pipeline_status=ROUTED. Trigger VaultHealthWorkflow for this employee. Publish DOC_ROUTED to Kafka → NotificationService sends employee alert.
