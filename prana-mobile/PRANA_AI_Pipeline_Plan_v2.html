End-to-End Flow

The 6-Stage Pipeline

OA selects a document type and uploads a batch. From that point, the pipeline takes over — no further human input required until (and unless) an exception is raised. Every NIK (PAN) value is encrypted at the ingestion boundary and never travels in plaintext beyond that point.

01
Batch Ingestion
File Handler · OCR
OA uploads ZIP or multi-file batch, declares doc type. System queues files for processing. OCR applied to scanned PDFs.
02
Encryption Boundary
FF3-1 FPE · HMAC · KMS
NIK extracted from document text, immediately encrypted. Plaintext NIK discarded. Only pan_token + enc_pan enter the system. LLM receives redacted doc.
03
Schema-Aware Extraction
LLM · Structured Output
LLM receives NIK-redacted doc + extraction schema. Returns JSON with non-NIK field values + confidence scores. NIK never sent to LLM in plaintext.
04
Identity Resolution
Token Match · Embeddings
pan_token matched against Employee Master token. Emp ID + fuzzy + embedding fallbacks for docs without NIK. All comparisons on encrypted/token values.
05
Tag & Key Assignment
Metadata Engine
Resolved identity, tenant, doc type, period written as immutable metadata. pan_token used as vault anchor. enc_pan + enc_dek stored. No plaintext NIK anywhere.
06
Vault Routing
Router · Notifier
Document delivered to employee's vault under correct org + doc-type bucket. Employee notified. Unresolved docs → Exception Queue.
Layer 1 is intentionally absent. The OA declares the document type at upload time — one doc type per batch. The pipeline does not need to classify; it starts with classification already known. This simplifies extraction and dramatically improves accuracy.
Identity Architecture

The NIK Model — National Identity Key

Rather than hardcoding PAN, PRANA abstracts the primary identity key as NIK — a country-configured field whose type varies by jurisdiction. The extraction engine and vault router always resolve to NIK. The country config tells the system what NIK maps to.

Country / Region NIK Field Characteristics
🇮🇳 India PAN 10-char alphanumeric. Lifetime. Appears on salary docs, Form-16.
🇺🇸 USA SSN Store last-4 only (privacy). Full SSN for matching internally.
🇦🇪 UAE Emirates ID 15-digit numeric. Biometric-linked.
🇬🇧 UK NI Number 9-char alphanumeric. Format: AA999999A.
🇸🇬 Singapore NRIC 9-char. S/T prefix for citizens/PRs.
🌐 Others Passport No. Fallback for countries without universal ID. Configurable per tenant.
Three Identity Layers

Vault Identity (NIK) — Permanent. Cross-org. Government-issued. PAN in India. Used for document routing across all tenants.

Login Credential (Mobile) — Authentication handle. Can change. Linked to vault via NIK. MFA (OTP + TOTP) layered on top.

Intra-Org Key (Emp ID) — Org-specific. Changes every employer. Used for intra-tenant routing only; never cross-org.

Mobile ≠ Identity Key. Mobile numbers change — people port SIMs, move countries. NIK (PAN) never changes. Mobile is the login handle; NIK is the vault anchor. If mobile changes, the employee updates it in profile — documents already routed via NIK are unaffected.
DPDP Compliance · Security Architecture

Three-Layer Encryption Architecture

Under India's Digital Personal Data Protection (DPDP) Act, PAN and equivalent NIK values cannot be stored in cleartext. PRANA implements a three-layer envelope encryption model — ensuring plaintext NIK never persists anywhere in the system, while enabling deterministic comparison for identity resolution and seamless cross-org vault continuity.

Core constraint: Standard AES encryption is non-deterministic — same plaintext produces different ciphertext each time (random IV). This breaks comparison. PRANA solves this with two separate mechanisms: HMAC for matching (deterministic) and AES-SIV / FF3-1 for storage (format-preserving, deterministic). Plaintext NIK is discarded the moment either is computed.
The Three Layers — What Each Does
Layer A
pan_token
HMAC-SHA256
Matching & Cross-Org Identity Anchor
HMAC-SHA256(PAN, platform_secret) — deterministic, always same output for same PAN. Used for all identity matching and vault routing. Platform-level secret (not tenant-level) ensures the same PAN from NPCI and ABCD Bank produces the same token — enabling cross-org vault linking. This is the vault anchor. Plaintext PAN never used after this point.
Layer B
enc_pan
AES-SIV / FF3-1 FPE
Secure Storage of Actual NIK Value
FF3-1_FPE(PAN, employee_DEK) — Format-Preserving Encryption, NIST standard. Encrypts PAN into a ciphertext of identical format and length (e.g. ABCDE1234F → XKQRT7823P). Encrypted with a per-employee Data Encryption Key (DEK). Used only when the actual PAN value must be retrieved (e.g. regulatory reporting). Never used for matching.
Layer C
enc_dek
Envelope Encryption · KMS
DEK Protection — Limits Blast Radius
KMS_Encrypt(DEK, tenant_KEK) — the per-employee DEK is itself encrypted by a tenant-level Key Encryption Key (KEK) that lives only in the KMS (AWS KMS / Azure Key Vault). KEK never touches the database. If a KEK is rotated, only enc_dek values need re-wrapping — the actual enc_pan values are untouched. One DEK compromise = one employee exposed, never the entire tenant.

What the Employee Master row looks like in the DB

-- employee_master table (per tenant) { "employee_uuid": "emp_9a2f7c...", // internal UUID — no PII "tenant_id": "npci_tenant_001", "pan_token": "hmac:a3f8c2...9d1e", // HMAC(PAN, platform_secret) — for matching "enc_pan": "fpe:XKQRT7823P", // FF3-1(PAN, emp_DEK) — for retrieval only "enc_dek": "kms:AQICAHg3...", // KMS_Encrypt(DEK, tenant_KEK) "employee_name": "Rahul Sharma", // name stored in clear (not PII under DPDP) "emp_id_org": "NPCI0042", "doj": "2019-03-01", "department": "Payments Infrastructure", "designation": "Sr. Systems Analyst" // NO plaintext PAN anywhere in this row }

Stage 02 — Encryption Boundary: What Happens in Memory

At the ingestion boundary — before any DB write, before LLM call
IN
Document text (post-OCR) containing visible PAN: "... PAN: ABCDE1234F ..."
STEP 1
Regex / pattern match extracts NIK from document text in memory. Plaintext NIK held in memory variable for <1ms.
STEP 2
Immediately compute both forms in memory:
pan_token = HMAC-SHA256("ABCDE1234F", platform_secret)
enc_pan = FF3-1_FPE("ABCDE1234F", employee_DEK)
STEP 3
Plaintext NIK variable zeroed and dereferenced from memory. GC eligible immediately.
STEP 4
Document text sent to LLM with NIK replaced by placeholder token: "... PAN: [NIK_REDACTED] ...". LLM never sees plaintext PAN.
OUT
Only pan_token and enc_pan flow downstream. All matching, resolution, tagging, and routing uses pan_token exclusively.

Cross-Org Vault Continuity — How It Works Without Decryption

// Rahul joins NPCI in 2019. NPCI uploads his Employee Master. NPCI → pan_token = HMAC("ABCDE1234F", platform_secret) = "tok_a3f8c2...9d1e" NPCI → enc_pan = FF3-1("ABCDE1234F", NPCI_DEK_rahul) = "XKQRT7823P" NPCI → enc_dek = KMS_Encrypt(NPCI_DEK_rahul, NPCI_KEK) // Rahul joins ABCD Bank in 2024. ABCD uploads his Employee Master. ABCD → pan_token = HMAC("ABCDE1234F", platform_secret) = "tok_a3f8c2...9d1e" ← SAME TOKEN ABCD → enc_pan = FF3-1("ABCDE1234F", ABCD_DEK_rahul) = "MNPVZ4917K" ← different DEK, different ciphertext ABCD → enc_dek = KMS_Encrypt(ABCD_DEK_rahul, ABCD_KEK) // Platform sees pan_token "tok_a3f8c2...9d1e" in both NPCI and ABCD tenants // → same person → documents from both orgs route to the same vault // → ABCD's KEK compromise does NOT affect NPCI's enc_pan (different DEK) // → platform_secret rotation re-computes all pan_tokens in one batch — no PAN decryption needed

Key Rotation — Zero Disruption

Rotate tenant KEK

Re-wrap all enc_dek values in that tenant with the new KEK.

enc_pan values untouched. pan_token untouched. Zero disruption to matching or routing.

Rotate employee DEK

Re-encrypt one employee's enc_pan with new DEK. Re-wrap new DEK with KEK.

Only that employee affected. All other employees unaffected. pan_token unchanged.

Rotate platform_secret

Re-compute all pan_token values across all tenants in a single batch job.

Requires temporary access to enc_pan + DEK for re-derivation. Scheduled maintenance window. All cross-org links preserved.

Compromise Blast Radius — What Each Breach Actually Exposes

What is compromised What attacker gets What they can do Blast radius Mitigation
One employee's DEK Can decrypt that employee's enc_pan → plaintext PAN Access one person's PAN only 1 EMPLOYEE Rotate that employee's DEK immediately. Re-encrypt enc_pan.
Tenant KEK Can decrypt all enc_dek values → all DEKs → all enc_pan in that tenant Access all PAN values for one org 1 TENANT Rotate KEK. Re-wrap all DEKs. Other tenants unaffected. pan_token unaffected.
Platform HMAC secret Can compute pan_tokens for known PANs (if attacker already has PAN list) Verify if specific PAN is in system. Cannot reverse pan_token → PAN. LIMITED Rotate platform_secret. Re-compute all pan_tokens. HMAC is one-way — PAN cannot be derived from token.
DB full dump (no keys) enc_pan ciphertexts, enc_dek blobs, pan_tokens Nothing useful. Cannot decrypt without DEKs. Cannot reverse HMAC tokens. ZERO Encryption at rest protects the full DB. This is the target architecture state.
Layer 2

Schema-Aware Field Extraction

The doc type is already known from the OA's declaration at upload. The LLM receives the document content plus a precise extraction schema — only the fields relevant to that doc type. It returns a structured JSON with field values and per-field confidence scores.

What the LLM receives

1. The full text of the document (post-OCR if scanned)

2. The extraction schema for the declared doc type — field names, expected formats, and examples

3. Instruction to return structured JSON only — no prose, no explanation

4. Instruction to assign a confidence score (0.0–1.0) to each extracted field

What the LLM returns

A structured JSON object with every field from the schema — populated where found, null where absent — plus a confidence score per field and an overall document confidence.

Fields with confidence below threshold (e.g., <0.75) are flagged for human review in the Exception Queue.

Sample LLM Extraction Prompt — Salary Slip
You are a document extraction engine. Extract ONLY the fields listed below from the provided salary slip document. Return a valid JSON object. Do not add any fields not listed. For each field, also return a confidence score between 0.0 and 1.0. DOCUMENT TYPE: Salary Slip DOCUMENT TEXT: {{ full_text_of_document }} EXTRACT THE FOLLOWING FIELDS: employee_id — Employee code / staff ID (e.g. NPCI001, EMP-1234) pan_number — PAN card number (format: AAAAA9999A) employee_name — Full name as printed on the document location — Office location / city gross_ctc — Gross CTC or total salary amount (numeric, INR) pf_number — PF account number uan_number — Universal Account Number (12-digit) tds_amount — TDS / income tax deducted (numeric, INR) pay_period_month — Month of the salary slip (e.g. March) pay_period_year — Year of the salary slip (e.g. 2024) RETURN FORMAT (JSON only, no prose): { "employee_id": { "value": "...", "confidence": 0.0 }, "pan_number": { "value": "...", "confidence": 0.0 }, "employee_name": { "value": "...", "confidence": 0.0 }, ... "overall_confidence": 0.0 }
Why per-field confidence matters: A salary slip might have a perfectly readable PAN but a smudged PF number (scanned doc). Per-field confidence lets the system route PAN-matched documents to the vault while flagging the PF field for OA review — rather than rejecting the whole document.

Layer 3

Resolution Priority

The Identity Resolution Priority Ladder

Resolution always attempts the highest-confidence key first and descends only if the key is absent or below confidence threshold. The ladder is universal — doc type determines which levels are reachable.

1
NIK — National Identity Key (PAN in India)
Strongest possible key. Government-issued, permanent, cross-org. Direct exact lookup in Employee Master NIK field. One match expected — if two employees share a PAN, that's a data quality error flagged immediately.
Confidence threshold: ≥ 0.85 · Doc types: Salary Slip, Form-16, Offer/Appointment Letter
2
Emp ID (within tenant)
Strong within a single org. Exact lookup scoped to the uploading tenant only — Emp ID NPCI0042 at NPCI is not the same as ABCD0042 at ABCD Bank. Used when PAN is absent (Increment, Promotion, Experience, Relieving letters).
Confidence threshold: exact match · Scope: current tenant only · Doc types: all letter types
3
Name + Date of Joining (Fuzzy)
Fuzzy string match on full name (Levenshtein distance ≤ 2 for names, exact or ±1 day for DOJ). Used when both NIK and Emp ID are absent. Handles minor OCR errors in name transcription. Requires both fields together — name alone is insufficient.
Confidence threshold: combined score ≥ 0.88 · Risk: common names (Singh, Kumar, Sharma) may match multiple employees
4
Embedding Similarity — Name + Dept + Designation
Generate a text embedding of the concatenated Name + Department + Designation string from the document. Compare cosine similarity against pre-computed embeddings of all Employee Master records. Match at cosine ≥ 0.92. Handles transliteration differences (Priya vs Priyaa), abbreviations (Mgr vs Manager), and dept naming variations.
Model: text-embedding-3-small or equivalent · Threshold: cosine ≥ 0.92 · Top-3 candidates returned for audit log
5
Exception Queue — Manual Resolution
Document could not be resolved to a unique employee with sufficient confidence. Routed to OA-Advance exception queue with all extracted fields, top candidate matches, and confidence scores displayed. OA-Advance manually assigns to correct employee. Assignment is logged and feeds back as a training signal.
Resolution time target: <24 hours · OA-Advance notified via dashboard alert + email

Exception Handling

The Exception Queue

Documents that cannot be auto-resolved land in the Exception Queue — visible only to OA-Advance users of the uploading tenant. The queue surfaces all extracted fields, top candidate matches with confidence scores, and a simple assignment interface.

No Match Found

No employee in the Master matches any extracted field at any threshold. OA-Advance must either assign manually or create a new Employee Master record and re-trigger resolution.

Multiple Candidates

Two or more employees match at similar confidence. System presents top-3 candidates with field-level match breakdown. OA-Advance selects correct employee. Selection is logged immutably.

Low Extraction Confidence

LLM extracted fields but overall confidence is below threshold (e.g. scanned doc with poor image quality). Document flagged for manual field review before resolution is attempted.

Feedback loop: Every manual resolution by OA-Advance is logged as a labelled training example. Over time, these examples can fine-tune the extraction model and improve embedding match thresholds — reducing the Exception Queue volume with each upload cycle.
Post-Resolution

Implementation

Recommended Tech Stack

OCR
Tesseract / AWS Textract
Scanned PDF → text. Textract preferred for structured forms (salary slips, Form-16).
Extraction LLM
GPT-4o / Claude Sonnet
Schema-aware field extraction with structured JSON output and per-field confidence scores.
Embeddings
text-embedding-3-small
Employee Master embeddings pre-computed and cached. Doc field embedding at resolution time.
Vector Store
pgvector / Pinecone
Store Employee Master embeddings per tenant. Scoped cosine similarity search at resolution.
Fuzzy Match
rapidfuzz
Python library. Levenshtein + Jaro-Winkler for name matching. Fast on 500–50k records.
Queue / Async
Celery + Redis
Async processing of bulk batches. Each document processed as an independent task. Retry on failure.
Storage
S3 + PostgreSQL
Files in S3 with tenant-prefixed keys. Metadata + routing tags in Postgres. pan_token indexed (never plaintext NIK).
Audit Log
Immutable Event Store
Every extraction, resolution, routing, exception, and manual override written to append-only log. No plaintext PAN in any log entry.
FPE / Encryption
FF3-1 · AES-SIV
Format-Preserving Encryption for NIK storage. Python: ff3 library. Java: Bouncy Castle. NIST SP 800-38G standard.
HMAC / Matching
HMAC-SHA256
Deterministic pan_token derivation. Platform-level secret in KMS. Python: hmac stdlib. One-way — PAN not derivable from token.
Key Management
AWS KMS / Azure Key Vault
Tenant KEKs and platform HMAC secret. Keys never leave HSM. Envelope encryption for per-employee DEKs. Rotation on schedule.
Processing speed estimate: For a batch of 500 salary slips (machine-generated PDFs, no OCR needed) — LLM extraction ~1–2s per doc, NIK exact match ~10ms per doc, tagging + routing ~50ms per doc. Full batch: approximately 15–20 minutes end-to-end at standard API rate limits, parallelised across a Celery worker pool.
